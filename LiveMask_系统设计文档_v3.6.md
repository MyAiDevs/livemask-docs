# LiveMask 系统设计文档 v3.6

> 本文档是当前架构基线。任何实现、重构、配置或接口变更必须先确认是否偏离本基线。

## 1. 系统边界

LiveMask 由以下核心仓库和层组成：

- `livemask-app`：客户端体验、登录、连接、支付入口、用户反馈、本地缓存
- `livemask-backend`：API、认证授权、业务状态机、配置下发、支付回调、审计
- `livemask-nodeagent`：节点侧配置拉取、状态上报、任务执行、降级模式
- `livemask-docs`：架构、规则、任务、接口和跨仓库影响的唯一真相来源
- Database：业务数据、配置数据、审计数据、迁移和约束

## 2. 四层闭环

任何用户可见功能必须形成闭环：

```text
App Client -> Backend API -> NodeAgent -> Database
     ^              |             |          |
     |              v             v          v
 用户反馈       状态机/配置      上报/降级    审计/约束
```

## 3. 配置原则

- 业务规则不得硬编码。
- 可变策略必须进入配置表、规则引擎或 FeatureFlag。
- 配置变化必须记录默认值、兼容策略、回滚策略和影响范围。
- NodeAgent 和 App 必须能处理旧配置、缺省配置和异常配置。

## 4. 接口原则

- API 变更必须说明请求字段、响应字段、错误码、状态流转和兼容策略。
- 删除或重命名字段前必须确认 App 与 NodeAgent 版本兼容。
- 支付、风控、权限、状态机接口必须具备幂等和审计。

## 5. 数据原则

- 新字段必须说明默认值、迁移方式、索引和约束。
- 状态变化必须可审计。
- 跨层任务必须能通过 `TASK-XXXX` 追踪。

## 6. 降级与回滚

每个高风险变更必须提供：

- 降级模式行为
- 回滚步骤
- 监控指标
- 人工介入入口
- 后续 TASK（如存在未完成项）

## 7. 架构变更流程

1. 在任务清单登记 `TASK-XXXX`。
2. 记录变更动机、影响范围和受影响仓库。
3. 更新本设计文档或对应子文档。
4. 完成实现后补充验证结果和回滚策略。
