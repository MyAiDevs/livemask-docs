## 6. 积分经济体系运营（2026-05-10 完善）

### 6.1 配置管理
- 所有 earning 规则（`node_points_rate`、`plan_purchase_bonus_rate`、`promoter_points_rate`）和 C2C 风控参数均通过 `system_configs.points_economy` 配置
- 修改后建议先在测试环境验证，并通过「影响模拟」查看对积分发行量的影响
- 大额交易 KYC 要求变更需同步更新用户协议和前端提示

### 6.2 日常监控重点
- C2C 市场异常交易（价格严重偏离、短时间内大量挂单/成交）
- 积分 earning 异常增长（可能存在节点刷流量或推广刷单）
- 积分在套餐购买中的使用比例（过高可能影响 USDT 收入）
- 平台从 C2C 交易中抽取的 USDT 收益

### 6.3 异常处理流程
1. 系统风控自动冻结异常交易 → 创建低优先级申诉工单
2. 运营人员在「积分交易管理」后台进行人工审核
3. 确认异常后执行积分回滚或交易取消，并通知双方用户
4. 严重刷单/洗钱行为可直接封禁账号 + 记录风控黑名单

### 6.4 与其他模块联动
- 积分 earning 异常 → 联动检查对应节点的质量评分和流量数据
- 大量积分用于购买套餐 → 关注套餐转化率和 LTV 变化
- C2C 交易量异常 → 可能存在洗钱风险，需加强风控审核

### 6.5 定期运营动作
- 每周查看「积分经济大盘」（总发行量、C2C 交易量、平台抽成收益、积分通胀率）
- 每月复盘 earning 规则是否合理，必要时调整参数

## 7. 多支付方式运营管理（2026-05 更新）

### 7.1 支付对账流程

**每日对账 Checklist**（值班人员每日必须执行）：

1. **USDT 支付对账**
   - 检查 NOWPayments / BTCPay 后台交易记录与系统 `payment_orders` 表是否一致
   - 重点关注 `pending` 状态超过 30 分钟的订单
   - 核对区块链浏览器交易确认数

2. **支付宝 / 微信支付对账**
   - 下载支付宝/微信商户平台对账单，与系统订单核对
   - 检查是否有 `paid` 但系统未更新的异常订单
   - 关注退款订单与系统退款记录是否一致

3. **Stripe（海外银行卡）对账**
   - 在 Stripe Dashboard 查看昨日交易
   - 检查 Webhook 是否全部成功接收
   - 关注 `incomplete` 状态的订阅

4. **Google Play / Apple IAP 对账**
   - 通过 Google Play Console / App Store Connect 查看订阅数据
   - 与系统 `external_subscriptions` 表进行比对
   - 重点检查取消订阅、退款、恢复订阅事件是否同步

**推荐做法**：开发每日自动对账任务，生成差异报告发送至运维 Telegram 群。

### 7.2 异常处理流程

**常见支付异常分类与处理**：

| 异常类型                   | 严重程度 | 处理流程                                                                 | 升级路径          |
|----------------------------|----------|--------------------------------------------------------------------------|-------------------|
| 支付回调验签失败           | P0       | 记录日志 → 人工核对 → 必要时联系支付平台                                 | SRE → 支付负责人  |
| 用户支付成功但系统未到账   | P0       | 立即人工补单 + 补偿积分/优惠券                                           | SRE → 客服        |
| 订阅自动续费失败           | P1       | 发送提醒 → 尝试重新扣款 → 进入 Dunning 流程                              | 客服              |
| 大额异常退款 / Chargeback  | P0       | 冻结相关账户 → 风控审核 → 必要时法务介入                                 | 风控 → 法务       |
| Google/Apple 订阅状态不同步| P1       | 触发恢复订阅接口 → 更新本地状态 → 通知用户                               | 后端开发          |
| 支付方式配置错误导致失败   | P1       | 立即下线该支付方式 → 检查配置 → 修复后重新上线                           | SRE               |

**处理原则**：
- 所有支付异常必须记录到 `payment_exception_logs` 表
- P0 级别异常必须在 15 分钟内响应
- 涉及资金的异常必须双人复核后处理

### 7.3 多支付方式切换与风控建议

- 新增支付方式上线前，必须通过 **Feature Flag** 进行灰度发布
- 不同支付方式的风控阈值应独立配置（例如 Stripe 大额交易风控更严格）
- 用户切换支付方式时，引导完成新支付方式绑定后再处理旧订阅

### 6.6 详细运营操作流程

#### 6.6.1 C2C 交易纠纷处理流程
1. 系统自动创建纠纷工单（当交易超过30分钟未完成或买家发起申诉）
2. 运营人员在「积分交易纠纷」后台查看证据（聊天记录、转账截图、系统日志）
3. 常见处理方式：
   - 买家胜诉：积分退还买家，USDT 退还卖家（平台承担损失或从保证金扣除）
   - 卖家胜诉：交易正常完成
   - 双方协商一致：按协商结果执行
4. 处理完成后系统自动同步积分余额和交易状态，并发送通知给双方

#### 6.6.2 异常 earning 监控与处理
- **节点刷流量 earning 过多**：联动节点质量评分系统，必要时降低该节点质量分或暂停 earning
- **推广大使刷单 earning**：检查被邀请用户是否为真实付费用户，异常时可回滚积分并警告大使
- **积分在套餐购买中使用比例过高**：可能影响 USDT 收入，需评估是否调整赠送比例

## 7. NodeAgent 降级模式运营处理指南（新增）

### 7.1 什么是降级模式
当 NodeAgent 检测到 sing-box 配置应用失败、频繁崩溃或 API 无响应时，会自动进入**降级保护模式**。
在此模式下：
- 节点会保持最后一次成功的配置继续运行（保护用户连接）
- 禁止自动应用新配置和自动重启
- 心跳中会携带 `degraded: true` 标记
- **只能由后台手动触发退出**

### 7.2 运营发现与处理流程
1. **告警接收**：收到 P0 告警「节点进入降级模式」
2. **查看详情**：在「节点监控大盘」或节点详情页查看 `degraded_reason` 和持续时长
3. **判断原因**：
   - 配置热更新失败 → 检查最新配置是否正确
   - 频繁崩溃 → 检查服务器资源、网络、sing-box 日志
4. **干预操作**：
   - 点击「强制重启 sing-box」
   - 或点击「强制退出降级模式」（系统会尝试拉取最新配置并应用）
5. **观察恢复**：等待 30~60 秒后查看心跳中 `degraded` 是否变为 false

### 7.3 常见处理建议
- 如果是配置问题：先在后台修正配置，再触发「强制退出降级模式」
- 如果是服务器问题：联系 Sponsor 排查资源或网络
- 如果频繁触发：考虑临时降低该节点质量分或暂时禁用

### 7.4 注意事项
- 降级模式下用户连接通常不受影响，但无法热更新配置
- 长时间处于降级模式的节点会影响 Sponsor 收益，需尽快处理
- 严禁在未确认原因前频繁重启 sing-box（可能加剧崩溃）

## 7. NodeAgent 运营与维护（新增）

### 7.1 NodeAgent 定位
NodeAgent 是部署在每个节点服务器上的守护进程，负责：
- 管理 sing-box 进程（启动、停止、热更新配置）
- 采集节点实时流量、系统资源、sing-box 统计数据
- 上报数据到后端（用于质量评分、流量监控大盘）
- 接收后端配置并应用到 sing-box

### 7.2 日常运营关注点
1. **NodeAgent 存活状态**：通过节点监控大盘或告警查看 Agent 是否在线。
2. **sing-box 进程状态**：Agent 会自动拉起异常退出的 sing-box，必要时人工介入。
3. **配置热更新失败**：查看 Agent 日志，确认是否因配置格式错误导致。
4. **数据上报延迟**：检查 Agent 与后端网络连通性，以及上报频率配置。

### 7.3 常见问题处理
- **NodeAgent 频繁重启**：检查资源限制（内存/CPU）、 sing-box 配置是否过大。
- **sing-box 无法启动**：检查生成的配置文件是否合法，可通过 `sing-box check -c config.json` 验证。
- **流量数据不准确**：确认 sing-box 的 `clash_api` 已开启，且 Agent 能正常访问 `127.0.0.1:9090`。
- **配置下发后未生效**：检查 Agent 是否成功收到配置（查看日志中的 `config applied`）。

### 7.4 权限与操作
- 超级管理员可远程触发节点配置刷新（通过后台操作）。
- 运营人员可查看 Agent 上报的详细 metrics（通过节点监控大盘）。
- 紧急情况下可通过 SSH 手动重启 NodeAgent 服务。

### 7.5 监控与告警
- NodeAgent 心跳丢失告警（P0）
- sing-box 进程异常退出告警（P1）
- 配置热更新失败告警（P1）
- 上报数据延迟过高告警（P2）
