# LiveMask AI辅助开发工作流与规范 v3.7（最终强化版）

**版本**：v3.7  
**更新日期**：2026-05-12  
**目的**：为 Cursor / Claude Code / Windsurf / Codex 等 AI 编码代理建立**最高效、可追溯、强闭环**的开发规则体系。  
**核心理念**：AI 不是“代码生成器”，而是“架构守护者 + 闭环执行者”。

---

## 1. 最高优先级铁律（AI 必须内化）

### 1.1 架构第一原则（禁止 feature creep）
在生成任何代码前，**必须先阅读并确认**以下文档是否支持该变更：
- `LiveMask_系统设计文档_v3.6.md`（整体架构与已打通闭环）
- `LiveMask_开发任务清单与里程碑_v3.6.md`（当前任务边界）
- `LiveMask_文档索引与关联说明_v3.6.md`（文档关联关系）

**禁止** 在未确认现有架构的情况下提出或实现新功能。

### 1.2 四大层闭环铁律（必须验证）
任何变更必须保持 **App Client + NodeAgent + Backend API + Database** 四层完整闭环：
- 数据流必须双向可追溯
- 异常必须有补偿机制（retry / rollback / degraded mode / notification）
- 配置变更必须通过 `system_configs` + 热更新 + 版本控制

### 1.3 Traceability 铁律（零容忍）
- 所有代码变更必须在文件头部或关键函数添加 `// TASK-XXXX: 说明`
- Commit Message 必须以 `[TASK-XXXX]` 开头
- 禁止在 `main` 分支直接开发
- 每个子任务必须完成 **开发 → 测试 → 文档同步 → 任务清单更新 → 合并** 完整闭环

## 2. AI 开发前必须执行的 Checklist（强制）

在开始任何编码前，AI 必须在思考中逐条确认：

1. [ ] 已阅读 `LiveMask_系统设计文档_v3.6.md` 相关章节，确认变更符合现有架构
2. [ ] 已确认该变更属于 `LiveMask_开发任务清单与里程碑_v3.6.md` 中的某个具体 Task
3. [ ] 已检查是否影响四大层闭环（App / NodeAgent / Backend / DB）
4. [ ] 业务逻辑是否可以通过 `system_configs` 配置化实现（禁止硬编码）
5. [ ] 是否涉及 NodeAgent（必须考虑 garble 混淆 + 单二进制 + 降级模式）
6. [ ] 是否涉及客户端（必须考虑 iOS/Android/Desktop 一致性 + 混淆 + 安全存储）
7. [ ] 是否涉及敏感操作（必须考虑审计日志 + 权限 + 回滚）
8. [ ] 变更后是否需要同步更新相关文档（系统设计 / API / 数据库 / 运营手册）
9. [ ] 是否需要更新多仓库 README（跨仓库变更时必须同步 `livemask-docs`）

## 3. 核心架构约束（AI 必须严格遵守）

### 3.1 后端约束
- 严格分层：Handler → Service → Repository
- 所有可变业务参数必须走 `system_configs` 表（JSONB）
- 定时任务统一使用 Asynq
- 通知统一走 Redis Streams + Worker
- 支付使用 `PaymentProvider` 工厂模式
- 留存预警使用 `RetentionRiskEngine` + 干预日志闭环
- 监控到行动使用 `AlertOrchestrator` + 自动决策

### 3.2 NodeAgent 约束（最高安全要求）
- 必须以 **单一 garble 混淆二进制** 形式交付
- 必须支持 Config hot update + 失败自动回滚 + degraded mode
- 诊断功能必须使用结构化指令 + 白名单（严禁直接执行 shell）
- 必须实现 mTLS + HMAC 签名通信
- 密钥必须通过 Vault 动态获取 + 定期轮换

### 3.3 客户端约束
- 所有智能决策放在 `ConnectionOrchestrator`
- sing-box 只负责传输，不包含业务逻辑
- iOS NetworkExtension 必须保持极简
- 必须实现 Certificate Pinning + 请求签名 + 防重放
- 本地敏感数据使用 `flutter_secure_storage`

### 3.4 数据库约束
- 核心业务表必须有 audit log / status 冗余字段
- 收益、流量、风险评分计算必须可追溯重算
- 禁止在代码中写死业务规则（全部通过配置或规则引擎）

## 4. 开发流程强制闭环（每个任务必须执行）

1. **领取任务** → 更新任务清单状态为「进行中」 + 填写当前分支
2. **AI 生成代码** → 必须引用 Task ID + 关联文档
3. **自检 DoD 清单**（见下方）
4. **开发者 Code Review**（关键逻辑必须人工 review）
5. **运行测试**（单元测试 + 必要集成测试）
6. **同步更新文档**（系统设计 / API / 数据库 / 运营手册）
7. **更新任务清单**（状态、Commit、测试结果、关联文件）
8. **标准 Commit**（使用模板）
9. **合并到 dev** + 删除分支
10. **标记任务为「已验证」**

**AI 必须在每一步主动提醒开发者执行以上流程。**

---

## 5. Definition of Done (DoD) 清单（AI 必须逐条自检）

完成任何子任务前，AI 必须在最终输出中确认以下全部通过：

- [ ] 代码中已添加 `// TASK-XXXX` 注释
- [ ] Commit Message 以 `[TASK-XXXX]` 开头
- [ ] 已更新 `LiveMask_开发任务清单与里程碑_v3.6.md` 中的所有 traceability 字段
- [ ] 核心逻辑已编写单元测试或集成测试（覆盖率要求见具体任务）
- [ ] 已同步更新相关技术文档
- [ ] 没有引入明显技术债务（或已在备注中记录）
- [ ] 配置相关变更已通过 `system_configs` 实现
- [ ] 客户端变更已考虑多平台一致性
- [ ] 安全相关变更已做自检
- [ ] 已按「任务完成汇报模板」输出完整内容
- [ ] 已验证四大层闭环完整性
- [ ] 跨仓库变更已同步更新 `livemask-docs` 仓库相关 README 和文档

---

## 6. 多仓库联动规则（强制）

- 中央文档仓库：`livemask-docs`（唯一真相来源）
- 所有仓库的 `README.md` 必须引用 `livemask-docs` 中的对应文档
- 跨仓库变更时，PR 描述必须注明「同步更新 livemask-docs」
- 推荐使用 GitHub Projects 作为跨仓库任务跟踪中心

---

## 7. 推荐的 AI 高效工作模式

1. **新对话开始时**：先让 AI 阅读本规则文件 + 当前任务的 Task ID + 关联文档
2. **复杂任务**：要求 AI 先输出「架构影响分析 + 闭环验证 checklist」
3. **代码生成后**：要求 AI 主动输出 DoD 自检结果 + 需要更新的文档列表
4. **每周复盘**：让 AI 读取任务清单，自动生成「本周完成情况 + 下周风险」

---

## 8. 禁止行为（零容忍）

- 禁止在未确认架构的情况下新增大功能
- 禁止硬编码业务规则（必须走配置或规则引擎）
- 禁止跳过 DoD 清单直接提交
- 禁止在 main/dev 分支直接开发
- 禁止不更新任务清单和文档就结束任务
